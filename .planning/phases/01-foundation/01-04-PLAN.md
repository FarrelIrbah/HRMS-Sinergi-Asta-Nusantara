<task type="auto">
  <name>Task 1: Extend Prisma client with automatic audit logging</name>
  <files>
    src/lib/prisma.ts
  </files>
  <action>
    Update `src/lib/prisma.ts` to add a Prisma Client Extension that automatically intercepts create, update, and delete operations and logs them to the AuditLog table.

    The approach: Use `$extends` with `query.$allModels` to intercept mutations. The challenge is that audit logging needs the current user's ID, which is not available in the Prisma context. There are two approaches:

    **Recommended approach -- Context-aware audit via service functions:**

    Since Prisma Client Extensions for `$allOperations` don't have access to the HTTP request context, the cleanest approach is:

    1. Keep the base Prisma client singleton as-is (globalThis pattern).

    2. Create a helper function `createAuditLog` that service functions call explicitly after mutations:
    ```typescript
    export async function createAuditLog(params: {
      userId: string
      action: AuditAction
      module: string
      targetId: string
      oldValue?: any
      newValue?: any
    }) {
      await prisma.auditLog.create({
        data: {
          userId: params.userId,
          action: params.action,
          module: params.module,
          targetId: params.targetId,
          oldValue: params.oldValue ?? undefined,
          newValue: params.newValue ?? undefined,
        }
      })
    }
    ```

    3. Service functions will call `createAuditLog` after each create/update/delete operation, passing the session user's ID. This is explicit, testable, and avoids the complexity of trying to thread user context through Prisma extensions.

    IMPORTANT: The `createAuditLog` function must NEVER be called recursively. It writes to AuditLog and does NOT trigger itself.

    Place the `createAuditLog` function in `src/lib/prisma.ts` alongside the Prisma singleton, since it's a core data-layer utility.
  </action>

    - The `createAuditLog` function is exported from `src/lib/prisma.ts`
    - No circular dependencies
  </verify>

<task type="auto">
  <name>Task 2: Create audit log query service</name>
  <files>
    src/lib/services/audit.service.ts

  <action>
    Create `src/lib/services/audit.service.ts` with functions to query the audit log:

    1. `getAuditLogs(filters)` -- paginated query with filters:
    - `userId` (optional string) -- filter by acting user
    - `module` (optional string) -- filter by module (e.g., "Manajemen Pengguna")
    - `action` (optional AuditAction) -- filter by action type
    - `dateFrom` (optional Date) -- filter from date
    - `dateTo` (optional Date) -- filter to date
    - `page` (number, default 1)
    - `pageSize` (number, default 25)
    - Returns: PaginatedResponse with AuditLog records including the `user` relation (name, email)
    - Order by: `createdAt` descending (newest first)
    - Use Prisma `where` with conditional spreading for optional filters:
      ```typescript
      where: {
        ...(userId && { userId }),
        ...(module && { module }),
        ...(action && { action }),
        ...(dateFrom || dateTo ? {
          createdAt: {
            ...(dateFrom && { gte: dateFrom }),
            ...(dateTo && { lte: dateTo }),
          }
        } : {}),
      }
      ```

    2. `getAuditLogById(id)` -- get a single audit log entry with full details:
    - Returns the AuditLog record with user relation
    - Used for the detail view showing full before/after diff
    - Returns null if not found

    3. `getAuditLogUsers()` -- get list of users who have audit log entries:
    - Returns distinct users from audit logs (for the user filter dropdown)
    - Returns: Array of { id: string, name: string, email: string }

    4. `getAuditLogModules()` -- get list of distinct modules:
    - Returns distinct module values from audit logs (for the module filter dropdown)
    - Returns: string[]

    All functions use the Prisma singleton from `src/lib/prisma.ts`.
    Import types from `@prisma/client` and `src/types/index.ts`.

    - Functions use proper Prisma query patterns (no raw SQL)
