<task type="auto">
  <name>Task 1: Create user validation schemas and service layer</name>
  <files>
    src/lib/validations/user.ts
    src/lib/services/user.service.ts
    src/lib/actions/user.actions.ts
  </files>
  <action>
    1. Create `src/lib/validations/user.ts`:
    - `createUserSchema`: name (string, min 2, max 100), email (string, email format), password (string, min 8, must contain uppercase + lowercase + number), role (enum Role)
    - `updateUserSchema`: name (string, min 2, max 100), email (string, email format), role (enum Role). NO password field (password change is separate or not in v1 scope).
    - Error messages in Indonesian: "Nama minimal 2 karakter", "Format email tidak valid", "Password minimal 8 karakter", "Password harus mengandung huruf besar, huruf kecil, dan angka"
    - Export TypeScript types: `CreateUserInput`, `UpdateUserInput`

    2. Create `src/lib/services/user.service.ts`:
    - `getUsers(params)`: Paginated user list. Params: page, pageSize, search (optional, searches name and email). Returns PaginatedResponse<User> (excluding hashedPassword). Order by createdAt desc.
    - `getUserById(id)`: Get single user by ID (excluding hashedPassword). Returns null if not found.
    - `createUser(data, actorId)`: Create user with hashed password (bcryptjs, 12 rounds). Check email uniqueness first -- if email exists, throw error "Email sudah terdaftar". After creating, call `createAuditLog` with module MODULES.USER, action CREATE, newValue (user data minus password).
    - `updateUser(id, data, actorId)`: Update user name, email, role. Check email uniqueness (excluding current user). Fetch old values BEFORE update for audit log. Call `createAuditLog` with oldValue and newValue.
    - `toggleUserActive(id, actorId)`: Toggle isActive boolean. Prevent deactivating yourself (compare id with actorId). Call `createAuditLog` with action UPDATE, old and new isActive values.
    - All service functions use the Prisma singleton from `src/lib/prisma.ts`.
    - Use `ServiceResult<T>` return type for error handling.

    3. Create `src/lib/actions/user.actions.ts` -- Server Actions:
    - Each action: check session via `auth()`, verify role is SUPER_ADMIN, validate input with Zod, call service function, revalidate path "/users".
    - `createUserAction(formData)`: Parse with createUserSchema, call createUser.
    - `updateUserAction(id, formData)`: Parse with updateUserSchema, call updateUser.
    - `toggleUserActiveAction(id)`: Call toggleUserActive.
    - Return `{ success: true }` or `{ success: false, error: "message" }`.
    - Use `"use server"` directive at top of file.
  </action>
  <verify>
    - `npx tsc --noEmit` passes
    - Validation schemas reject invalid inputs (test mentally: empty name, short password, invalid email)
    - Service functions handle all edge cases (duplicate email, self-deactivation)
  </verify>
  <done>
    User validation, service layer, and server actions complete. CRUD operations create audit log entries. Email uniqueness enforced. Self-deactivation prevented.
  </done>
